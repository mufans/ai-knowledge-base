# 技术内容深度分析总结技能

## 使用场景

- 已完成技术内容采集（如 github-trending），需要对原始数据做深度分析
- 需要从一批项目中提炼技术亮点、评估价值、发现趋势
- 为知识库产出结构化的分析报告，辅助技术决策

## 执行步骤

### 步骤 1：读取最新采集文件

使用 Glob 扫描 `knowledge/raw/` 目录，找到最新的采集文件：

```
knowledge/raw/github-trending-YYYY-MM-DD.json
```

选取文件名中日期最近的一个，使用 Read 读取其完整内容。

### 步骤 2：逐条深度分析

对每个项目进行以下四个维度的分析：

**摘要**（不超过 50 字）

- 一句话概括项目的核心定位和最大亮点
- 去掉修饰词，只保留关键事实
- 示例：`> 面向 RAG 场景的混合检索引擎，原生支持多模态文档解析和向量化。`

**技术亮点**（2-3 个）

- 用事实说话，引用具体技术特征而非主观评价
- 可从项目 README、架构设计、性能指标中提炼
- 示例格式：
  ```
  - 采用 GraphRAG 架构，支持知识图谱 + 向量双路召回
  - 单节点支持百万级文档索引，查询延迟 < 100ms
  - 原生集成 LangChain / LlamaIndex，5 分钟完成接入
  ```

**评分**（1-10 分）及理由

评分标准：

| 分数区间 | 含义 | 判定依据 |
|---------|------|---------|
| 9-10 | 改变格局 | 开创全新范式、解决行业级痛点、已被广泛验证 |
| 7-8 | 直接有帮助 | 可立即用于生产、解决明确工程问题、社区活跃度高 |
| 5-6 | 值得了解 | 有参考价值但需观望、处于早期阶段、适用场景较窄 |
| 1-4 | 可略过 | 同质化严重、缺乏文档、长期未维护 |

评分后必须附上一句理由，例如：`8分 - 已被多家企业用于生产环境，文档完善，可直接落地。`

**标签建议**

为项目建议 2-4 个分类标签，用于后续知识库归类，例如：`["RAG", "向量检索", "开源框架"]`。

### 步骤 3：趋势发现

在逐条分析完成后，纵览所有项目，提炼：

**共同主题**

- 多个项目反复出现的共通方向或模式
- 例如："多个项目都在解决 Agent 记忆管理问题，说明这已成为工程落地的关键瓶颈"

**新概念**

- 首次出现或近期才兴起的技术概念、架构模式、工具类型
- 例如："Model Context Protocol (MCP) 作为 AI Agent 的标准化工具调用协议，正在被越来越多的框架采纳"

### 步骤 4：输出分析结果 JSON

将完整的分析结果写入 `knowledge/analysis/tech-summary-YYYY-MM-DD.json`，日期与原始采集文件保持一致。

## 约束条件

- 分析的项目数以输入文件的实际 items 数量为准，不做截断
- **9-10 分项目不超过 2 个**，严控高分名额，确保评分有区分度
- 摘要严格控制在 50 字以内
- 技术亮点必须有具体事实支撑，禁用"非常强大"、"十分优秀"等空洞表述
- 如果项目信息不足以判断，宁可给低分并标注"信息不足"

## 注意事项

- 确保输出目录 `knowledge/analysis/` 存在，不存在则先创建
- JSON 文件使用 UTF-8 编码，确保中文内容正确输出
- 分析完成后告知用户分析的项目数量、评分分布和文件路径
- 趋势发现部分应基于项目事实归纳，不要凭空推测
